# ALLUR

A mobile-first AI fitness transformation coach MVP (brand: **ALLUR**, design theme: **TITANIUM** — "Strong. Modern. Refined.") — onboarding, goal-based training plans, progress photos, PR tracking, a simulated voice coach, and a 3-tier subscription system (Free / Base $12.99 / Premium $29.99). Users sign up with email/username/password (real persisted accounts); each athlete's durable fitness data (profile, goal, plan, progress photos, PRs, weight logs, meals, physique analysis, credits) is now persisted per-user to the backend and survives logout. Chat history stays ephemeral (session-scoped, intentionally not persisted).

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` — run the API server (port 5000)
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
- Required env: `DATABASE_URL` — Postgres connection string

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- API: Express 5
- DB: PostgreSQL + Drizzle ORM
- Validation: Zod (`zod/v4`), `drizzle-zod`
- API codegen: Orval (from OpenAPI spec)
- Build: esbuild (CJS bundle)

## Where things live

- **Training knowledge base**: `artifacts/fitcoach/src/data/trainingKnowledge.ts` — the research-backed source of truth for program generation. Encodes per-goal philosophy, splits by experience level, rep/rest/volume targets, cardio + protein guidance, and target-physique emphasis days. `buildProgram(profile, goal)` returns `{ days: Workout[], meta: ProgramMeta }`; consumed by onboarding's `generatePlan` and the admin demo seed, with `programMeta` rendered as a summary card on the Plan page.
- **Weekly volume model** (Jeff Nippard's evidence-based set guidelines): `VOLUME_GUIDELINES` in `trainingKnowledge.ts` encodes per-experience weekly-set landmarks (Beginner MEV 1–5 / growth 4–12; Intermediate MEV 4–8 / growth 8–15; Advanced MEV 4–8 / growth 12–20 / specialization 15–25), per-muscle weekly sweet spots, a progression model by experience, the diminishing-returns rule, and the hard-set intensity rule (0–3 RIR). `volumeTargetFor(experience)` builds the summary string. `buildProgram` populates `ProgramMeta.volumeTarget`, `setQuality`, `volumeDiminishingReturns`, and `perMuscleVolume`; the Plan page renders these as the "Weekly Set Target" card and the "Hard Sets Per Muscle / Week" section. The same numbers are mirrored server-side in `fitnessKnowledge.ts` (`VOLUME_RULES`, injected into `FITNESS_KNOWLEDGE_REFERENCE`) so the coach reasons with the identical framework — when editing the volume model, update both files in lockstep.
- App state/types: `artifacts/fitcoach/src/context/FitCoachContext.tsx` (React state hydrated from / persisted to the backend per-user, `Goal`/`UserProfile`/`Workout` types, macro/protein constants).

## Architecture decisions

- The frontend (`artifacts/fitcoach`) keeps all user data in `FitCoachContext`. The durable slice (`PersistedFitCoachState`: onboarding flag, profile, goal, credits, plan, programMeta, PRs, weight logs, progress photos, meals, physique analysis) is persisted per-user; chat history is excluded (ephemeral). Several features call the backend: the AI Coach, Physique Analysis, Meal Analysis (all three see Product), Admin mode, and the new per-user state store.
- **Per-user persistence** (`GET`/`PUT /api/me/fitness-state`, `artifacts/api-server/src/routes/me.ts`): both endpoints are auth-gated (`req.isAuthenticated()`, keyed by `req.user.id` — no client-supplied id, so no IDOR). State is a single JSONB blob in `userFitnessStateTable` (`lib/db/src/schema/fitness.ts`, `userId` PK FK→`users.id` cascade); PUT validates the body with the generated `SaveMyFitnessStateBody` zod schema then upserts via `onConflictDoUpdate`. Progress photos are downscaled base64 data URLs, so they serialize into JSONB directly (no object storage). `FitCoachContext` hydrates once per logged-in user (`useGetMyFitnessState`, query key scoped by userId, `staleTime: Infinity`) and debounces writes (800ms, `useSaveMyFitnessState`). Guards: (1) the debounced writer never fires until the initial read has settled (`hydratedUserIdRef`), so empty defaults can't clobber existing server data, and a failed read leaves persistence disabled rather than overwriting; (2) on logout or in-session account switch the `hydrated` flag flips false and `AuthGate` shows a spinner until the new user's state loads, so one account's in-memory data is never rendered under another; (3) entering the admin demo sets `adminTaintedRef`, disabling persistence so the throwaway seed never overwrites a real account — `exitAdminMode` and logout clear the taint and force a fresh re-hydrate.
- **AI Coach** lives in `artifacts/api-server/src/routes/coach.ts`. `runCoach()` is the shared text+voice brain (OpenAI chat completions + an `update_training_plan` tool); the system prompt is built by `lib/coachPrompt.ts` (training knowledge re-encoded as plain data — the React `trainingKnowledge.ts` can't be imported server-side). Endpoints are unauthenticated by design (no user accounts) but protected by an in-memory per-IP rate limiter.
- **Physique Analysis** (`POST /api/coach/analyze-physique`, same file) is a real OpenAI vision call: it compares the user's photo to two embedded reference body-fat charts (`lib/bodyFatReferenceImages.ts`) plus a text knowledge base (`lib/bodyFatKnowledge.ts`) and returns a structured estimate (body-fat range, confidence, visual markers, limitations, suggested direction, per-muscle ratings) via a forced `report_body_fat_analysis` tool. It has its own stricter rate-limit bucket + MIME/size guard since vision is the priciest call. `BODY_FAT_COACH_REFERENCE` is also injected into the text coach prompt so both features stay consistent.
- **Meal Analysis** (`POST /api/coach/analyze-meal`, same file) is a real OpenAI vision call too: it identifies the meal in a food photo and returns name, visible items, and an approximate calorie/macro breakdown with a confidence level via a forced `report_meal_analysis` tool (sets `isFood=false` for non-food photos → 400). It shares the `visionRateLimit` bucket + the same MIME/size guard as physique analysis. The Macros page (`/macros`) consumes it.
- **Coach knowledge base**: `lib/fitnessKnowledge.ts` exports `FITNESS_KNOWLEDGE_REFERENCE` — a condensed, research-backed KB (training principles, pain/injury handling, exercise substitution by movement pattern, equipment, progression/deloads, nutrition + macro engine, dietary restrictions, supplements, log-adaptation, program assignment). It is injected into the text coach system prompt by `lib/coachPrompt.ts`. Like `coachPrompt.ts`, it is a server-side plain-data re-encoding (the React `trainingKnowledge.ts` can't be imported server-side).
- **Subscription tiers (Free / Base / Premium)**: server is the source of truth. `lib/credits.ts` defines `UserPlan = "free"|"base"|"premium"` and `BASE_MONTHLY_CREDITS` (coaching 50 / photo 150 / bodyScan 20 — also mirrored as display-only copy in the client `artifacts/fitcoach/src/lib/subscription.ts`, keep both in lockstep); `spendCredit` is free→deny / base→decrement / premium→unlimited. `lib/creditGuard.ts` returns 403 `{type:"needs_subscription"}` for free, 402 for base-exhausted, passes premium — **this is the authoritative enforcement**; the client paywall/locks are UX only and may fail-open on infra failure. `lib/stripe/plan.ts` maps a Stripe subscription → product `metadata.plan` (premium>base>free; untagged legacy→premium) and exposes `getSubscriptionSummary(userId)`. Routes: `POST /api/stripe/checkout {plan}` (base → `trial_period_days:14` on first sub + `payment_method_collection:"always"`), `POST /api/stripe/cancel` (`cancel_at_period_end`), `GET /api/me/subscription`. Client: `FitCoachContext` derives `plan`/`isSubscribed`/`isPremium` from `/me/subscription` first (falls back to `/me/credits`, then free) so a transient credits failure or post-checkout sync lag can't falsely lock a paying user. Brand-new users are forced to `Paywall` after onboarding (`App.tsx` RouteGuard, keyed on `subscription.hasEverSubscribed`); lapsed users drop to Free (data kept) and hit `LockedFeature` (Coach, Macros) or `needsSubscriptionToast` (Plan AI-adjust, Start Workout, Progress physique scan) with resubscribe CTAs. Cancel/upgrade/resubscribe all live in `Account.tsx`.
- **Admin mode + admin allowlist** also use the backend. `lib/admin.ts` is the single source of truth: an admin is the Repl owner (`REPL_OWNER_ID`) OR an email on the `ADMIN_EMAILS` env var (comma-separated; read at module load, so **changing it requires an API-server restart**). `GET /api/admin/status` returns `{ isAdmin, isOwner }` (`isAdmin = isOwner || allowlisted`); the `/admin` console is gated on `isAdmin`. Admins are also treated as **Premium everywhere**: `credits.ts:getUserPlan` returns `"premium"` (unlimited, never decremented) and `stripe/plan.ts:getSubscriptionSummary` returns a synthetic `{plan:"premium",status:"active",hasEverSubscribed:true}` so admins get full unlimited access and the `hasEverSubscribed` flag bypasses the post-onboarding paywall — no client change needed (the client derives plan/paywall from `/me/subscription`). An allowlisted email only grants admin to the account that already owns that email (the `users` unique-email constraint blocks a second account from claiming it).
- **Credential auth** (custom email/username/password accounts) coexists with Replit OIDC in the **same** `users` table (`artifacts/api-server/src/routes/auth.ts`). `POST /api/auth/register` / `/login` / `/logout` use bcryptjs and create sessions with no `access_token`/`expires_at` (so `refreshIfExpired` leaves them untouched). The table has nullable `username` (unique) + `password_hash` columns so OIDC-only rows still validate. Two coexistence guards: (1) usernames are constrained to exclude `@` (`pattern` in the OpenAPI `RegisterAccountRequest`) and login resolves `@`-containing identifiers as email, others as username — so the email/username namespaces never overlap and a login can't match an ambiguous pair; (2) OIDC `upsertUser` links to an existing row by **email** when one exists under a different id (a prior credential signup), instead of inserting and tripping the unique-email constraint — otherwise OIDC login would 500 for that person. Both auth endpoints share a per-IP rate limiter (10/min) to blunt password guessing. The web app gates everything behind `AuthGate` in `App.tsx` (shows the `/auth` page when unauthenticated), except `/admin`, which is bypassed so the OIDC owner-preview flow still works.
- Auth (sessions/users tables, OIDC routes) lives in `artifacts/api-server`; the web auth hook is `@workspace/replit-auth-web` (`useAuth`). The fitcoach web app reads auth via `AuthContext` (`useAccount`, backed by the generated `useGetCurrentAuthUser` hook).

## Product

- **Marketing landing** (`/home`): a full-width marketing page for signed-out visitors (hero, problem/solution, how-it-works, objections, pricing, FAQ) in the TITANIUM aesthetic (`artifacts/fitcoach/src/pages/landing/Landing.tsx`, framer-motion + lucide-react, bespoke imagery in `public/`). Pricing reflects the real tiers (Base $12.99 with a 14-day trial CTA "Start 14-Day Free Trial"; Premium $29.99 with "Get Premium" — no trial on Premium). CTAs route to `/auth?mode=signup` / `?mode=login`. Routing lives in `App.tsx` `AuthGate`: signed-out users see `<Landing/>` at `/home` and `<Auth/>` at `/auth` (everything else redirects to `/home`); signed-in users on `/home` or `/auth` redirect to `/`. `Auth.tsx` reads the initial form view from the `?mode` URL param (wouter strips the query, so `location === "/auth"` still matches) and its Back button returns to `/home`.
- **AI Coach** (`/coach`): a real conversational coach (not scripted). Users chat by text or push-to-talk voice. It answers training questions grounded in the knowledge base, holds a back-and-forth, and when the user agrees to a concrete change it **automatically applies it to their plan** (the Plan page updates immediately). Chat/plan/profile state is session-scoped in `FitCoachContext` — no DB persistence for chat.
- Backed by OpenAI (via the Replit AI integration): text on `POST /api/coach/chat`, voice on `POST /api/coach/voice` (STT → same coaching brain → TTS audio reply).
- **Physique Analysis** (`/progress`): when the user uploads a progress photo and taps "Analyze My Physique", the photo is downscaled client-side and sent to the vision endpoint. The result renders as a body-fat range with a confidence badge, the visual markers the AI saw, a per-muscle development breakdown, a recommended direction, and an accuracy caveat — replacing the old fake/seeded analysis. (The owner-only admin demo still uses a synthetic `buildPhysiqueAnalysis` to preview the screen without a real upload.)
- **Meal logging** (`/macros`): the user snaps/uploads a meal photo; it is downscaled client-side and sent to `/api/coach/analyze-meal`. The real vision result (name, items, calories, macros) is logged into the day's macro counter — replacing the old random `SAMPLE_MEALS` simulation.
- **Plan adjustments** (`/plan`): the "Request AI Adjustment" dialog now POSTs to `/api/coach/chat` (same brain as the Coach page) and applies the returned `updatedPlan` immediately — previously a toast-only no-op.
- **Dashboard** body-fat stat reads the real `physiqueAnalysis.bodyFatEstimate` (shows `—` until a physique analysis exists) — previously a hardcoded `14.5%`.
- **Onboarding guidelines** (`/` step 5): two sections — "Any injuries the Coach should know about?" and "Any dietary restrictions?" — each supports chip options + free-form text + a voice note. Voice notes are transcribed via `POST /api/coach/transcribe` (STT-only) and appended to the notes field. Stored on `UserProfile` as `injuries`/`injuryNotes`/`dietary`/`dietaryNotes`; `composeGuideline(chips, notes)` joins them into a single guideline string sent to the coach. The server prompt treats them as hard constraints (`guidelinesSection` in `coachPrompt.ts`). After onboarding, a loading screen genuinely awaits plan generation: `buildProgram` runs first, then (only if injuries are present) `POST /api/coach/chat` adapts the plan to train safely around them; navigation happens only after this resolves. If adaptation fails, the user keeps the base plan and gets a toast telling them to ask the Coach.

## Gotchas

- The API server runs behind Replit's reverse proxy, so `app.set("trust proxy", true)` is required for `req.ip` to reflect the real client. Without it, the coach per-IP rate limiter would key every request to the single proxy IP and throttle all users together.
- Voice STT must detect the audio container from the bytes (`detectAudioFormat`), not the client-declared format — browsers record different containers (Chrome webm, Safari mp4). Anything not directly transcribable is converted to wav via ffmpeg (`ensureCompatibleFormat`).
- The OpenAI integration proxy does **not** support `POST /audio/speech`; TTS goes through `gpt-audio` chat completions with `modalities: ["text","audio"]` (this is what the `textToSpeech` helper does).
