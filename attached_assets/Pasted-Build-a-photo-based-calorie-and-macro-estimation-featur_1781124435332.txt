Build a photo-based calorie and macro estimation feature for my fitness app.

The feature should allow a user to upload or take a photo of a meal. The app should analyze the image, identify individual food items, estimate portion sizes, ask clarification questions when confidence is low, then calculate calories, protein, carbs, and fat using a nutrition database.

The app should not pretend that the image alone can perfectly determine macros. The image model should identify foods and estimate portions. Nutrition numbers should come from a database such as USDA FoodData Central for generic foods, Open Food Facts for packaged/barcoded foods, and an internal common fitness food database for fast fallback.

The goal is to create a CAL.AI-style experience where the user can quickly scan a meal, get a strong macro estimate, and easily correct anything the AI is unsure about.

FEATURE GOAL

Allow users to upload or take a photo of a meal and receive an estimated breakdown of:

* Calories
* Protein
* Carbs
* Fat
* Optional: fiber, sugar, sodium
* Food items detected
* Estimated serving sizes
* Confidence level
* Editable meal log

The feature should prioritize speed, usability, and reasonable accuracy. The app should always let the user correct uncertain foods or portion sizes.

CORE USER FLOW

1. User uploads or takes a photo of their meal.
2. AI analyzes the image.
3. AI detects individual food items.
4. AI assigns a confidence score to each item.
5. AI estimates portion size for each item.
6. App checks food items against a nutrition database.
7. App calculates calories and macros.
8. App displays a meal summary.
9. If confidence is low, app asks the user a clarification question.
10. User confirms, edits, or corrects the result.
11. Meal is saved to the user’s food log.
12. Corrections are stored to improve future estimates for that user.

CONFIDENCE RULES

Use this confidence threshold system:

High confidence: 0.80 or higher

Show result without asking first.

Example:

“Looks like chicken breast, white rice, and broccoli.”

The user should still be able to edit the result.

Medium confidence: 0.55 to 0.79

Show a confirmation question before finalizing.

Example:

“I think this is chicken, but it could be salmon. Which one is it?”

Low confidence: below 0.55

Do not finalize macros yet. Ask the user what the food is.

Example:

“I’m not sure what the main protein is. Is it chicken, salmon, steak, tofu, or something else?”

PORTION ESTIMATION SYSTEM

Portion size is the hardest part. The MVP should use heuristic estimation first.

Estimate food amount using:

* Food area relative to plate
* Number of visible pieces
* Standard plate size assumptions
* Hand-size comparisons if hands or utensils are visible
* Common serving size defaults
* User’s past confirmed meals
* Food category density
* Whether the food appears flat, piled, sliced, liquid, sauced, or mixed

MACRO CALCULATION FORMULA

Once the food and serving size are known, calculate:

food_macros = nutrition_per_100g * estimated_grams / 100

Example:

{
"food": "chicken breast cooked",
"estimated_grams": 150,
"per_100g": {
"calories": 165,
"protein": 31,
"carbs": 0,
"fat": 3.6
},
"calculated": {
"calories": 247.5,
"protein": 46.5,
"carbs": 0,
"fat": 5.4
}
}

Then sum all foods to create the total meal estimate.

You are a food recognition and nutrition estimation assistant inside a fitness tracking app.

Analyze the uploaded meal image.

Your job is to identify each visible food item, estimate the likely portion size, and return structured JSON.

Do not invent certainty. If you are unsure whether an item is chicken, salmon, pork, tofu, rice, pasta, potatoes, sauce, or something else, include the top possible alternatives and lower the confidence score.

Important rules:

RECOMMENDED FEATURE BEHAVIOR

Do this:

* Always let users edit results.
* Always separate food identification confidence from portion confidence.
* Ask clarifying questions for uncertain proteins, carbs, sauces, and mixed meals.
* Prefer database lookup over AI memory.
* Save user corrections.
* Show that results are estimates.
* Use barcode scan for packaged foods.
* Let users create custom foods.
* Let users save repeat meals.
* Let users add missing items.
* Let users mark whether they ate all, half, or part of the meal.

Do not do this:

* Do not pretend image macros are exact.
* Do not rely only on the vision model for nutrition facts.
* Do not guess between chicken and salmon when confidence is low.
* Do not ignore oils, sauces, dressings, and butter.
* Do not force the user through too many questions.
* Do not ask for clarification on every tiny item.
* Do not over-prioritize vegetables or garnishes that barely affect macros.

FINAL VIBE CODER INSTRUCTIONS

Build this as a photo-based meal scanner and macro estimator.

The system should use image analysis to detect foods, not to directly determine nutrition numbers.

Nutrition should come from a food database.

The system should:

1. Analyze uploaded meal images.
2. Detect visible foods.
3. Estimate portions.
4. Check confidence.
5. Ask the user questions when confidence is low.
6. Use a nutrition database to calculate macros.
7. Let users edit every item.
8. Save the meal to their log.
9. Save corrections to improve future results.

The most important feature is the clarification flow.

If the app cannot confidently tell whether something is chicken, salmon, steak, pork, tofu, rice, pasta, potato, sauce, or something else, it must ask the user before calculating final macros.

The app should make the process feel fast and easy, but it should never pretend to be perfectly accurate.

Use language like:

“Estimated calories”

“Estimated macros”

“Looks like…”

“Confirm this item”

“I need one quick check to make this more accurate.”

Do not use language like:

“Exact calories”

“Perfect macro count”

“Guaranteed accurate”

Best MVP stack:

1. Vision model for food detection.
2. USDA FoodData Central for generic nutrition.
3. Open Food Facts for branded/barcoded foods.
4. Internal common fitness food database.
5. Clarifying questions when confidence is low.
6. User corrections to personalize future estimates.

This will create a strong CAL.AI-style experience without needing to train a massive custom food recognition model from scratch.
